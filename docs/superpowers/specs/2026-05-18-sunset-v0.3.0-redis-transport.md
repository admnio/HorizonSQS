# Sunset v0.3.0 — Redis Transport

**Date:** 2026-05-18
**Status:** Draft (awaiting user review)
**Package:** `admnio/sunset` (extending v0.2.0)
**Sub-project of:** Sunset for Laravel — a Horizon replacement

## Roadmap Context

Sunset is a planned multi-release Horizon replacement. This spec covers **only v0.3.0**. Roadmap:

| Version | Scope |
|---|---|
| v0.2.0 (shipped) | Rename `horizon-sqs` → Sunset + `Transport` interface + SQS transport |
| **v0.3.0 (this spec)** | Sunset's own Redis transport implementing the `Transport` contract |
| v0.4.0 | `sunset:work` supervisor replacing `php artisan horizon` |
| v0.5.0 | Sunset repositories replacing Horizon's |
| v1.0.0 | Full SPA dashboard, drops `laravel/horizon` dependency |
| v1.1.0 | Worker CPU/Memory monitoring |
| v1.2.0 | Realtime worker activity stream |
| v1.3.0 | Queue pause/resume controls |

Each row is its own spec → plan → implementation cycle.

## Goal

Add a `RedisTransport` so users with Redis-backed Laravel queues get the same Sunset/Horizon dashboard parity that SQS users got in v0.2.0. Drops into existing Redis queue configs (`'driver' => 'redis'`) with zero migration friction.

## Acceptance Criteria

1. New tree exists: `src/Transports/Redis/{RedisConnector,RedisQueue,RedisTransport}.php`.
2. `TransportRegistry` resolves `'redis'` to `RedisTransport` (registered in `SunsetServiceProvider::register()` alongside `SqsTransport`).
3. `Queue::connection('redis')->push(...)` flows through Sunset's `RedisQueue`, dispatches Horizon's `JobPending`/`JobPushed`/`JobReserved` events, and the job appears in Horizon's `/horizon/api/jobs/pending` and `/horizon/api/stats.recentJobs`.
4. All pre-existing 56 tests still pass; new tests cover the Redis transport (~6 unit + ~2 integration).
5. README + UPGRADING.md mention v0.3.0 Redis support.

## Non-Goals (Deferred)

- Replacing Horizon's `RedisJobRepository`, `RedisMetricsRepository`, etc. (v0.5.0)
- Custom dashboard pages for Redis-specific views (v1.0.0)
- Cluster/sentinel-specific connection logic — Laravel's `Illuminate\Contracts\Redis\Factory` already handles this; we delegate
- BC shims for users running Horizon-only Redis without Sunset — those users install Sunset and restart workers; no code-level shim needed
- Forking Laravel's `RedisQueue` — we extend, not replace

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Parent class for `RedisQueue` | Extend `Illuminate\Queue\RedisQueue`, dispatch Horizon events ourselves | Same pattern as `SqsQueue`. Sunset never imports Horizon's queue classes — only its event types. No coupling to Horizon's Redis internals. |
| Driver registration | `Queue::extend('redis', ...)` — override Laravel's stock connector | Same pattern as SQS. Load-order priority means Sunset wins over Horizon's `redis` override since Sunset's provider boots after Horizon's. |
| Connection management | Delegate to `Illuminate\Contracts\Redis\Factory` (Laravel's standard) | Cluster/sentinel/phpredis-vs-predis decisions stay in the user's `config/database.php`. Sunset stays connection-agnostic. |
| Workload data source | `LLEN queues:{name}` + `ZCARD queues:{name}:delayed` + `ZCARD queues:{name}:reserved`, summed | Mirrors Horizon's existing schema (Sunset's RedisQueue inherits Laravel's, which uses these same keys). |
| Error handling on workload | Per-queue try/catch, log warning, length=0 fallback | Identical to `SqsTransport::workload()` — one bad queue can't blank the whole dashboard. |
| Enrichment | Use Horizon's `JobPayload::prepare()` inside `pushRaw` | Same pattern as `SqsQueue`. No new enrichment code. |
| `RedisConnector` shape | Thin shim delegating to `TransportRegistry::get('redis')->connect()` | Identical pattern to `SqsConnector` (v0.2.0). |

```php
    }

```

### `RedisTransport`

```php
namespace Admnio\Sunset\Transports\Redis;

use Admnio\Sunset\Contracts\Transport;
use Illuminate\Contracts\Queue\Queue;
use Illuminate\Contracts\Redis\Factory as RedisFactory;
use Psr\Log\LoggerInterface;
use Psr\Log\NullLogger;
use Throwable;

class RedisTransport implements Transport
{
    public function __construct(
        private RedisFactory $redis,
        private array $packageConfig,
        private ?LoggerInterface $logger = null,
    ) {}

    public function name(): string { return 'redis'; }

    public function connect(array $config): Queue
    {
        return new RedisQueue(
            redis: $this->redis,
            default: $config['queue'] ?? 'default',
            connection: $config['connection'] ?? 'default',
            retryAfter: $config['retry_after'] ?? 60,
            blockFor: $config['block_for'] ?? null,
            dispatchAfterCommit: $config['after_commit'] ?? null,
            migrationBatchSize: $config['migration_batch_size'] ?? -1,
        );
    }

    public function workload(array $queues): array
    {
        $logger = $this->logger ?? new NullLogger();
        $connectionName = $this->packageConfig['transports']['redis']['workload_connection'] ?? 'default';
        $conn = $this->redis->connection($connectionName);

        $records = [];
        foreach ($queues as $queue) {
            try {
                $length = (int) $conn->llen("queues:{$queue}");
                $length += (int) $conn->zcard("queues:{$queue}:delayed");
                $length += (int) $conn->zcard("queues:{$queue}:reserved");
            } catch (Throwable $e) {
                $logger->warning('sunset: Redis workload query failed for queue', [
                    'queue' => $queue,
                    'error' => $e->getMessage(),
                ]);
                $length = 0;
            }

            $records[] = [
                'name' => $queue,
                'length' => $length,
                'wait' => 0,         // filled by SunsetWorkloadRepository
                'processes' => 0,    // filled by SunsetWorkloadRepository
                'split_queues' => null,
            ];
        }

        return $records;
    }
}
```

### `RedisQueue`

Extends `Illuminate\Queue\RedisQueue`. Mirrors the event-dispatching pattern of `SqsQueue`. Key methods:

```php
namespace Admnio\Sunset\Transports\Redis;

use Illuminate\Queue\RedisQueue as LaravelRedisQueue;
use Laravel\Horizon\Events\JobPending;
use Laravel\Horizon\Events\JobPushed;
use Laravel\Horizon\Events\JobReserved;
use Laravel\Horizon\JobPayload;
use Illuminate\Support\Str;

class RedisQueue extends LaravelRedisQueue
{
    private mixed $lastPushed = null;

    public function push($job, $data = '', $queue = null)
    {
        $this->lastPushed = $job;
        return parent::push($job, $data, $queue);
    }

    public function pushRaw($payload, $queue = null, array $options = [])
    {
        $payload = (new JobPayload($payload))->prepare($this->lastPushed);

        $this->raiseHorizonEvent(new JobPending($payload->value), $queue);

        $result = parent::pushRaw($payload->value, $queue, $options);

        $this->raiseHorizonEvent(new JobPushed($payload->value), $queue);

        return $payload->id();
    }

    public function later($delay, $job, $data = '', $queue = null)
    {
        $this->lastPushed = $job;
        return parent::later($delay, $job, $data, $queue);
    }

    public function pop($queue = null)
    {
        $job = parent::pop($queue);

        if ($job !== null) {
            $this->raiseHorizonEvent(new JobReserved($job->getReservedJob()), $queue);
        }

        return $job;
    }

    private function raiseHorizonEvent($event, ?string $queue): void
    {
        $queueName = Str::replaceFirst('queues:', '', $queue ?? $this->default);
        event($event->connection($this->getConnectionName())->queue($queueName));
    }
}
```

Caveat: `parent::pop()` for Redis returns `RedisJob`, which has `getReservedJob()`. Verify the SDK shape during implementation; adapt event constructor args if needed.

## Migration

### From Sunset v0.2.0 (SQS only) to v0.3.0

Nothing breaks. SQS-only users were never using `'driver' => 'redis'`. If they add a Redis-backed queue connection later, it'll automatically use Sunset's Redis transport.

### From Horizon-only Redis (no Sunset) to v0.3.0

1. `composer require admnio/sunset`
2. `php artisan vendor:publish --tag=sunset-config`
3. No `config/queue.php` changes — existing `'driver' => 'redis'` blocks Just Work.
4. No `config/horizon.php` changes.
5. Restart workers (`php artisan horizon:terminate`, then start).

Sunset's connector overrides via load order. Same Redis schema, same event types, same dashboard.

### Mixed SQS + Redis

Both transports run side-by-side. Each `Queue::connection(...)` resolves to the appropriate transport. Horizon dashboard reads from both.

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Horizon's own `redis` driver registration races with ours | Sunset's `boot()` runs after Horizon's via Composer dependency order. Verified working for the same pattern with `sqs` in v0.2.0. |
| Laravel's `RedisQueue::pop()` returns `RedisJob` with `getReservedJob()` shape we assume | Verify against vendor during implementation. Adapt if SDK shape differs. |
| Users on cluster/sentinel Redis | We delegate to `Illuminate\Contracts\Redis\Factory` which handles these via `config/database.php`. No additional work. |
| The `workload_connection` config defaults to `'default'` — users with multi-Redis setups may need to override | Documented in config comment + UPGRADING. |
| `parent::pushRaw` accepts third arg `$options` whose meaning differs from SQS; passing through unchanged | Confirmed correct — Laravel's `RedisQueue::pushRaw` accepts `array $options = []` (currently only honors `delay`). Pass-through is safe. |
