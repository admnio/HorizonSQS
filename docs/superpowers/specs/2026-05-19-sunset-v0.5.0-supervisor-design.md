# Sunset v0.5.0 Design Spec ‚Äî Supervisor Subsystem

**Date:** 2026-05-19
**Status:** Approved for implementation planning
**Predecessor:** v0.4.0 (job lifecycle subsystem) ‚Äî merged 2026-05-19

## Goal

Move the supervisor + master + worker process tree out of `laravel/horizon`
and into the `Admnio\Sunset` namespace. After v0.5.0, no class under
`src/Supervisor/`, `src/SupervisorCommands/`, `src/MasterSupervisorCommands/`,
`src/Repositories/Redis/`, or `src/Console/Sunset*` references
`Laravel\Horizon\*` outside of the existing `Admnio\Sunset\Adapters\Horizon\`
boundary. `php artisan horizon` is removed (replaced with a 410-gone shim);
`sunset:work` is the new entry point.

## Non-goals (deferred to later releases)

- **Sunset SPA dashboard + dropping `laravel/horizon` from composer** ‚Äî
  deferred to **v1.0.0**.
- **Worker CPU/memory monitoring** ‚Äî v1.1.0.
- **Realtime worker activity stream** ‚Äî v1.2.0.
- **Queue pause/resume controls (dashboard-side)** ‚Äî v1.3.0. (CLI-side
  `sunset:pause`/`sunset:continue` ship in v0.5.0; the dashboard UI for
  per-queue pause is what's deferred.)

v0.5.0 ships strictly the supervisor subsystem. `laravel/horizon` stays as
a composer dependency (the dashboard reads our data through adapters).

## Decisions locked during brainstorming

1. **Clean break on the user-facing surface.** `sunset:work` requires a
   `config/sunset.php` `supervisors` block; reading `config/horizon.php`'s
   `environments` is not supported. `php artisan horizon` is replaced by a
   stub that prints a removal notice and exits 1. A
   `sunset:migrate-horizon-config` artisan command copies existing
   `config/horizon.php` `environments` to the new shape.

2. **Full CLI surface.** All four categories the user picked ship in v0.5.0:
   - Process tree commands: `sunset:work`, `sunset:supervise`, `sunset:worker`
   - Control commands: `sunset:pause`, `sunset:continue`,
     `sunset:pause-supervisor`, `sunset:continue-supervisor`,
     `sunset:terminate`, `sunset:status`, `sunset:supervisors`,
     `sunset:supervisor-status`
   - Maintenance commands: `sunset:clear`, `sunset:purge`,
     `sunset:snapshot`, `sunset:forget-failed`
   - Operator commands: `sunset:install`, `sunset:publish`
   - Migration: `sunset:migrate-horizon-config`
   - Plus the override: `horizon` ‚Üí `SunsetHorizonRemovedCommand`

3. **Approach: aggressive port.** Horizon's supervisor logic
   (`MasterSupervisor`, `Supervisor`, `ProcessPool`, `SupervisorProcess`,
   `ProvisioningPlan`, `SupervisorOptions`, `SupervisorFactory`,
   `ListensForSignals`) is ported line-by-line with namespace and naming
   changes only. Same balance strategies (`off`/`simple`/`auto`), same
   auto-scaling math, same signal handling, same process model. No
   structural redesign ‚Äî that belongs to a later release.

4. **Dashboard compat via adapters.** The four supervisor-related Horizon
   contracts (`MasterSupervisorRepository`, `SupervisorRepository`,
   `ProcessRepository`, `HorizonCommandQueue`) get bound to Sunset adapters
   in `SunsetServiceProvider`. The dashboard's pause/resume/restart buttons
   continue to work transparently.

## Architecture

```
sunset:work command (master process ‚Äî long-running)
  ‚îÇ
  ‚ñº
SunsetMasterSupervisor (PID=N)
  ‚îÇ persists state ‚Üí Sunset MasterSupervisorRepository (sunset:masters:{name})
  ‚îÇ reads ProvisioningPlan from config('sunset.supervisors')
  ‚îÇ pushes AddSupervisor commands onto our SupervisorCommandQueue
  ‚îÇ main loop: processSignals ‚Üí processCommands ‚Üí monitorSupervisors ‚Üí persist
  ‚îÇ
  ‚ñº spawns one per supervisor via Symfony Process
  SupervisorProcess (subprocess) ‚Äî runs:
    `php artisan sunset:supervise <name> <connection> --balance=... --max-processes=...`
    ‚îÇ
    ‚ñº inside that subprocess
    SunsetSupervisor (PID=M, parent=N)
      ‚îÇ persists state ‚Üí Sunset SupervisorRepository (sunset:supervisors:{name})
      ‚îÇ owns ProcessPool(s) ‚Äî one per queue when balancing, one total when not
      ‚îÇ reads SupervisorCommandQueue for pause/continue/scale/restart/terminate
      ‚îÇ main loop: processSignals ‚Üí processCommands ‚Üí balance/scale ‚Üí monitorProcesses ‚Üí persist
      ‚îÇ
      ‚ñº spawns one per pool slot via Symfony Process
      WorkerProcess ‚Äî runs:
        `php artisan sunset:worker <connection> --name=... --supervisor=... --queue=... --max-jobs=... --memory=128`
        ‚îÇ
        ‚îî extends Laravel's queue:work, dispatches Sunset lifecycle events
          (already exist from v0.4.0)
```

**OS support:** POSIX/pcntl required at runtime (same as Horizon). Tests for
process-spawning behavior run in Docker on Linux. Windows dev runs repo +
command-queue tests; integration tests for the supervisor tree are
Linux-only (guarded by `PHP_OS_FAMILY !== 'Linux' ‚Üí markTestSkipped`).

### Boundary rules

- **`src/Supervisor/`, `src/SupervisorCommands/`, `src/MasterSupervisorCommands/`,
  `src/Repositories/Redis/`, `src/Console/Sunset*Command.php`** ‚Äî no
  `use Laravel\Horizon\*` imports. Verified by grep.
- **`src/Adapters/Horizon/`** ‚Äî the only place Horizon contracts get
  imported. When v1.0.0 drops the Horizon dep, this entire directory is
  deleted.
- **pcntl-only at runtime.** The supervisor classes import `pcntl_*` and
  `Symfony\Component\Process\Process`. Unit tests for the process-spawning
  behaviors mock the Process abstraction; integration tests run on Linux.

the namespace differs.

## Redis key layout

`config('sunset.key_prefix')` (introduced in v0.4.0) prefixes everything.

| Sunset key                                         | Used for                              | Horizon equivalent                              |
|----------------------------------------------------|---------------------------------------|--------------------------------------------------|
| `sunset:masters`                                   | set of master supervisor names        | `horizon:masters`                                |
| `sunset:master:{name}`                             | per-master hash (status/PID/etc.)     | `horizon:master:{name}`                          |
| `sunset:supervisors`                               | set of all supervisor names           | `horizon:supervisors`                            |
| `sunset:supervisor:{name}`                         | per-supervisor hash                   | `horizon:supervisor:{name}`                      |
| `sunset:supervisor:{name}:orphans`                 | orphan PID tracking (TTL)             | `horizon:supervisor:{name}:orphans`              |
| `sunset:commands:{queue-name}`                     | command queue for control commands    | `horizon:commands:{queue-name}`                  |
| `sunset:master:{name}:command-queue`               | master's command queue                | `horizon:master:{name}:command-queue`            |

## Supervisor process model + signal handling

Ported from Horizon line-by-line:

- **Pcntl handling** via `ListensForSignals` trait. SIGINT/SIGTERM ‚Üí
  `terminate(0)`. SIGTSTP ‚Üí `pause()`. SIGUSR2 ‚Üí `continue()`. SIGCONT ‚Üí
  `continue()`.
- **Subprocess spawning** via `Symfony\Component\Process\Process`. Command
  lines built by `SupervisorCommandString` (master ‚Üí supervisor) and
  `WorkerCommandString` (supervisor ‚Üí worker). The `--parent-id={pid}`
  flag is passed so subprocesses can detect orphan state.
- **Termination cascade:** `sunset:terminate` (or SIGTERM) tells the master
  to terminate. Master pushes Terminate command to each supervisor's
  queue. Each supervisor scales its pool to 0 (workers drain in-flight
  jobs up to their `timeout`). Master waits up to
  `SupervisorRepository::longestActiveTimeout()` seconds before exiting.
- **Auto-scaling** via `Supervisor::balance()` and `ProcessPool::scale()`.
  Three strategies kept:
  - `off` ‚Äî no scaling; worker count is `maxProcesses`.
  - `simple` ‚Äî balance work across queues with equal weight.
  - `auto` ‚Äî auto-scale based on queue depth (`size` strategy) or expected
    time-to-clear (`time` strategy).

`MasterSupervisor` and `Supervisor` implement `Pausable + Restartable +
Terminable`.

### `horizon` command override mechanism

`SunsetHorizonRemovedCommand` declares `protected $signature = 'horizon
{--environment= : The environment name}';`. Since our provider boots before
Horizon's (alphabetical: `A` < `L`), Laravel's command registry resolves
`horizon` to our class. Horizon's own `HorizonCommand` is never reached.

```php
class SunsetHorizonRemovedCommand extends Command
{
    protected $signature = 'horizon {--environment=}';
    protected $description = '[Removed in Sunset v0.5.0 ‚Äî use sunset:work]';

    public function handle(): int
    {
        $this->components->error('The "horizon" command has been removed in admnio/sunset v0.5.0.');
        $this->components->bulletList([
            'Use "php artisan sunset:work" to start the supervisor.',
            'See UPGRADING.md for the v0.4.0 ‚Üí v0.5.0 migration steps.',
        ]);
        return 1;
    }
}
```

## `sunset:migrate-horizon-config` command

Signature:

```
sunset:migrate-horizon-config
    {--dry-run : Print what would be written without modifying any file}
    {--force : Overwrite existing supervisors block in config/sunset.php}
```

Algorithm:

1. Load `config/horizon.php` via `require` (the file is a return-array).
2. Extract `environments`, `memory_limit`, `fast_termination`, `wait`.
3. Load `config/sunset.php`. If a `supervisors` key exists and `--force` is
   not set, error out.
4. Construct the new `supervisors` block. Direct copy of `environments`.
   Pass-through `memory_limit`, `fast_termination`, `wait` keys.
5. If `--dry-run`: print the resulting PHP array as a code block.
6. Otherwise: rewrite `config/sunset.php` with the new keys inserted (use
   the existing config file as a template ‚Äî preserve comments and existing
   `transports`/`trim`/etc. via a targeted regex insertion before the
   closing `];`).

Output a summary: which keys were copied, line count delta.

     supervisor's queue name.

     `app(SupervisorCommandQueue::class)->pending(...)`.
  4. (For v0.5.0 we don't run a real supervisor in this test ‚Äî we just
     verify the routing. End-to-end is covered by `SupervisorLifecycleTest`.)

## Open implementation questions (resolve during plan)

- **`Symfony\Component\Process\Process` vs `proc_open`:** Horizon uses
  `Symfony\Component\Process\Process` (it's already a Laravel dep). We do
  the same. No proc_open direct calls.
- **`ListensForSignals` trait location:** the trait uses `pcntl_*` directly.
  On Windows where pcntl is unavailable, instantiating a class that uses
  the trait would fail. Solution: guard the trait's setup method with
  `function_exists('pcntl_async_signals')`. Same trick Horizon uses.
- **`SunsetWorkerCommand` vs Laravel's `queue:work`:** Horizon's
  `WorkCommand` extends `Illuminate\Queue\Console\WorkCommand` and adds an
  `ignore_user_abort` call. We do the same. The Sunset lifecycle events
  fire from our transports (Task 15 of v0.4.0 already updated them).
- **`SunsetSuperviseCommand` (the supervisor subprocess) signature:**
  matches Horizon's `horizon:supervisor` exactly ‚Äî all the same options
  (`--balance`, `--max-processes`, `--min-processes`, `--memory`,
  `--timeout`, etc.). The flag names are stable across the port.
- **`horizon` command override fragility:** if a future Horizon release
  registers the `horizon` command with higher priority (it currently uses
  `#[AsCommand(name: 'horizon')]` attribute, same as ours would), our
  override may break. A `test_horizon_artisan_command_is_overridden`
  regression test catches this.
